# 英语秒读压缩训练系统（Sentence Compression System）
## 产品需求文档 V2.0（终稿）

---

### 1. 产品定义与战略锚点

**产品名称**：英语秒读压缩训练系统

**一句话价值主张**：用“删词训练”让阅读速度提升2-3倍

**产品本质**：**阅读信息压缩训练工具**

**核心判定**：
本产品不是英语学习系统，不是语法训练系统，不是提分课程。
它的本质是**训练人如何快速阅读信息的工具**。

**核心差异**：
本产品不教英语，而是训练一种可量化的行为——**“只看核心信息，不读干扰信息”**。这使得本产品避开了英语教学的竞争红海，切入“信息处理效率”这一普适、可量化、高价值赛道。

### 3. H5核心页面设计

#### PAGE 1：入口页
**定位**：极简钩子，3秒传达价值

- **标题**：❗90%学生英语慢，是因为不会“删词读句子”
- **副标题**：用3分钟测试你的阅读压缩能力
- **行动按钮**：👉 开始训练

---

#### PAGE 2：训练页（核心交互）
**定位**：产品核心行为触发地

- **示例句子**：
  ```
  The boy who was running in the park yesterday suddenly stopped because he saw something strange.
  ```

- **用户操作（两态切换设计）**：
  - **点击词** → 在「正常黑色」与「灰字+删除线」之间切换
  - **默认状态**：所有词初始为正常黑色（用户“保留”）
  - **点击后**：该词变为灰字+删除线（用户标记为“可删除的干扰信息”）
  - **再次点击同一词**：恢复为正常黑色（撤销操作）

- **训练模式标识**：
  - **正常训练**：按难度顺序从句子库抽取
  - **针对性训练**：从指定知识点标签的句子中随机抽取3-5句
  - **两种模式在提交后的跳转路径不同**：
    - 正常训练 → PAGE 3 → PAGE 3.5 → PAGE 4
    - 针对性训练 → PAGE 3 → PAGE 3.5（更新错题聚合）

- **交互设计目的**：
  - 降低操作难度，避免误删恐惧
  - 最终视觉结果：仅核心主干为黑字，其余全部灰字+删除线，形成强烈视觉反馈：“你只需读黑字，速度就上来了”
  - **重置按钮**：一键恢复所有词为黑色

- **界面元素**：
  - 句子展示区（可点击交互）
  - 重置按钮
  - 提交按钮
  - 计时器（不显眼但存在，用于后端速度计算）

---

#### PAGE 3：标准答案对比页
**定位**：结果可视化，让用户看到“差距”

- **左侧**：用户的版本
- **右侧**：系统标准压缩版本
- **差异分析**：
  ```
  你删除了：
  ✔ 修饰信息  ✔ 时间信息  ✔ 地点信息

  你保留了：
  ✔ 主干结构  ✔ 动作核心
  ```

- **视觉规则**：
  - 用户正确删除的干扰词：绿色高亮
  - 用户错误删除的核心词：红色高亮
  - 用户遗漏未删的干扰词：黄色高亮

#### PAGE 3.5：错题聚合回顾页（行为层）
**定位**：将多轮训练中的错误按知识点聚合，转化为“可复习的错题集”，强化薄弱点突破

**角色说明**：PAGE 3.5 负责展示“错了哪些题”+“再来一组同类型训练”（行为层），PAGE 4 负责解释“为什么错”+“知识点讲解”（认知层）。用户先看“错在哪”，再看“为什么错”。

**触发时机**：
- **免费体验**：完成1句体验后展示（错题只有1句，简版）
- **付费7天训练**：每天完成当日全部句子后展示（跨多句聚合）
- **训练首页**：提供“我的错题本”入口，随时查看历史累计错题

**错题聚合规则**：
- 按知识点标签（tags）分组：定语从句 / 状语从句连词 / 介词短语 / 时间状语 / 短语动词 等
- 同一知识点下，按错误次数从高到低排序
- 每道错题展示三个信息：原文句子、用户操作、正确做法

**UI示意（付费用户每日训练后）**：
```

│                                      │
│  ┌─ 定语从句（错2题）──────────────┐  │
```

**“针对性训练”交互（关键闭环）**：

- 系统从句子库中筛选与错题知识点（如“定语从句”）同标签的句子
- 生成3-5句一组的新训练，专门针对该薄弱点
- 训练流程进入 PAGE 2（针对性训练模式），完成后回到 PAGE 3.5，形成闭环

**设计目的**：
- 不依赖 LLM，纯规则引擎聚合，零额外成本
- 将“做错”转化为“针对性再练”，形成完整的训练闭环
- 付费用户每天看到自己的薄弱点在收窄，产生正向激励
- 错题本作为训练首页的常驻入口，提升回访频次

### 4. 核心功能模块与算法逻辑

#### 模块一：句子训练模块
- **句子库规模**：100-300句（MVP阶段）
- **难度分级**：简单 / 中等 / 高考真题
- **核心交互**：点击词在两种状态间切换：
  - 正常黑色 = 用户保留的核心信息
  - 灰字+删除线 = 用户标记为可删除的干扰信息

#### 模块二：压缩对比引擎
- **标准压缩版本生成规则**：
  - **MVP阶段采用人工标注**：每个句子由运营人员在录入时手动标注标准压缩版本（保留词索引）。100-300句的标注工作量约2-3天。
  - **后续可引入AI辅助**：使用LLM生成建议压缩版本，人工审核后入库，效率可提升5-10倍。
  - **不建议MVP阶段自研规则引擎**：英语语法边界情况极多（短语动词如"look after"中的after易被误删，限制性定语从句可能改变语义），纯规则引擎在MVP阶段开发周期反而比人工标注长2-3倍。

- **标准压缩标注规则（人工标注遵循）**：
  1. 保留主干（主谓宾/主系表）—— 必须保留
  2. 删除非限制性修饰语（介词短语、非限制性定语从句等）
  3. 限制性定语从句视情况保留（如"the boy who is wearing a red hat"中，去掉"who is wearing a red hat"则不知道是哪个男孩）
  4. 状语从句标记逻辑连接词保留（because, unless, although, if等），其余部分可压缩
  5. 短语动词整体保留（look after, give up, put up with等）

- **差异计算算法**：
  - `删除正确率` = (用户正确删除的干扰词数 / 系统标准删除词数) × 100%
  - `核心保留率` = (用户保留且正确的核心词数 / 系统标准核心词数) × 100%

#### 模块三：速度评分系统
- **核心指标**：`阅读压缩指数 = (核心保留率 × 删除正确率) / 用时`
  - 两个指标**相乘**而非加权：防止用户通过“全删光”或“什么都不删”来刷分
  - 任一指标过低都会拉低总分，激励用户真正理解句子结构
- **展示形式**：
  - 今日 vs 昨日对比
  - 7天趋势曲线
- **设计原则**：不展示传统分数，只展示可量化的行为效率变化

#### 模块四：训练闭环系统（含 LLM 反馈 + 错题回顾）
```
输入句子 → 删词交互 → 压缩对比 → 错题聚合回顾 + 能力评分 + LLM反馈 → 行为记录 → 次日难度升级 → 针对性训练
```
- **难度自适应逻辑**：
  - 连续3次核心保留率 > 90% → 句子难度升级
  - 连续2次核心保留率 < 60% → 句子难度降级或同级别巩固

#### 模块五：LLM 智能反馈引擎

**定位**：基于每次训练的错误模式，借助大语言模型生成个性化的知识点诊断与复习建议，将“训练数据”转化为“可学习的知识点反馈”。

**触发时机**：
- 每次提交训练结果后，后端异步调用 LLM 生成反馈
- LLM 反馈生成后缓存到数据库，下次查看时直接读取（避免重复调用浪费费用）

**LLM 输入——结构化 Prompt 设计**

系统将训练数据组装为结构化请求发送给 LLM：
```json
{
  "sentence": "The boy who was running in the park yesterday suddenly stopped because he saw something strange.",
  "user_deleted": ["who", "was", "running", "in", "the", "park", "yesterday", "because"],
  "correct_deletion": ["who", "was", "running", "in", "the", "park", "yesterday"],
  "core_words": ["The", "boy", "suddenly", "stopped", "he", "saw", "something", "strange"],
  "errors": [
    {"type": "false_delete", "word": "because", "explanation": "保留逻辑连接词，because是状语从句关键连接词"},
    {"type": "missed_delete", "word": "running", "explanation": "running在定语从句中，属于从句谓语"}
  ],
  "sentence_tags": ["定语从句", "状语从句", "介词短语", "时间状语"],
  "user_level": "中等",
  "history_weak_tags": ["定语从句", "状语从句"]
}
```

**LLM 输出——结构化反馈**：
```json
{
  "weak_points": [
    {
      "tag": "定语从句",
      "severity": "high",
      "feedback": "你漏删了定语从句部分\"who was running in the park\"。定语从句修饰名词，在阅读压缩时通常可以整体删除，只保留被修饰的名词即可。",
      "missed_words": ["who", "was", "running"]
    },
    {
      "tag": "状语从句逻辑连接词",
      "severity": "medium",
      "feedback": "你删掉了\"because\"，但逻辑连接词（because, although, unless, if等）应该保留，它们是句子逻辑关系的骨架。",
      "false_deleted_words": ["because"]
    },
    {
      "tag": "时间状语识别",
      "severity": "low",
      "feedback": "你正确删除了\"yesterday\"，做得很好。时间状语是典型的可压缩信息。",
      "is_praise": true
    }
  ],
  "overall_assessment": "你对介词短语的识别不错，但定语从句是你目前的薄弱环节。连续3句都漏删了定语从句部分。",
  "review_suggestion": "建议复习：定语从句的基本结构（关系代词who/which/that引导的从句），练习识别从句的边界。推荐教材章节：高中英语定语从句专题 P.82-90",
  "next_focus": "定语从句"
}
```

**反馈分级策略**：
| 用户类型 | 反馈详略 | LLM调用策略 |
|----------|----------|-------------|
| 免费体验（匿名） | 仅显示1-2条最关键的反馈 | 预置模板（非LLM调用，省成本） |
| 免费体验（已授权） | 2-3条反馈，含1条复习建议 | LLM调用，结果缓存24小时 |
| 付费用户 | 完整反馈+历史累计薄弱点分析 | LLM调用，结果永久缓存 |

**成本控制**：
- 单次 LLM 调用预估：输入 ~200 tokens，输出 ~300 tokens
- 使用低成本模型（如 DeepSeek / Qwen 等国产模型 API）
- **缓存策略**：同一用户 + 同一句子 + 同一次训练 → 反馈结果缓存到数据库，二次查看不重复调用
- **聚合策略**：付费用户每天最多触发 3 次 LLM 调用（每次提交后），其余训练使用规则模板 + 历史缓存
- **成本估算**：假设单价 ¥1/百万 tokens，每次训练约 ¥0.0005，日均3次 × 付费用户数，10万付费用户日均成本约 ¥15

**降级策略**：
- LLM API超时（>5s）或返回错误 → 回退到规则模板
- 规则模板：按 error_tags 匹配预设的通用反馈文案（如“定语从句”对应固定话术）
- 前端展示时不区分LLM生成/模板生成，用户无感知

#### 模块六：错题聚合回顾引擎

**定位**：纯规则引擎，零 LLM 成本，将用户的错误训练记录按知识点自动聚合，生成可复习、可再练的错题集，驱动“犯错→针对性训练→减少犯错”的正向闭环。

**聚合逻辑**：
```
从 training_records 中查询最近 N 条 error_tags 非空的记录
    ↓
按 error_tags 分组，统计每个标签的出现频次
    ↓
每个标签下，按错误类型归类：
  ├─ missed_delete（应删未删）→ 展示“你漏掉了这些干扰词”
  └─ false_delete（不应删却删了）→ 展示“这些核心词不该删”
    ↓
按错误频率降序排列，取 Top 5 知识点
    ↓
从 sentences 表查询对应句子的原文和标准压缩信息
    ↓
渲染为错题回顾卡片
```

**“针对性训练”逻辑**：
```
用户点击“再来一组针对性训练”
    ↓
取当前错题聚合中频率最高的知识点标签（如“定语从句”）
    ↓
从 sentences 库中筛选包含该标签且用户尚未做过的句子
    ↓
随机取 3-5 句，组成本轮“针对性训练”
    ↓
进入 PAGE 2（针对性训练模式）
    ↓
完成后回到 PAGE 3.5，重新聚合——该知识点的错误数应减少
```

**数据支撑**：
- 依赖 training_records 表中的 `errors` 和 `error_tags` 字段
- 依赖 sentences 表中的 `tags` 字段（筛选同标签句子）
- 无需新增数据表，无需 LLM 调用

**展示时机**：
| 场景 | 展示内容 | 触发方式 |
|------|----------|----------|
| 免费体验 | 简版错题卡片（1句） | 完成训练后自动展示 |
| 付费每日训练 | 当日错题聚合（多句） | 当日全部句子完成后展示 |
| 训练首页 | “我的错题本”入口 | 常驻入口，累计所有历史错题 |
| 每日首次打开 | “你还有3个薄弱知识点未攻克” | 推送/首页横幅提醒 |

**与模块五（LLM反馈）的关系**：
- 模块六（错题回顾）与模块五（LLM反馈）**互补而非替代**
- 模块五负责“解释为什么错”，给出知识点讲解和复习建议（偏“理解”）
- 模块六负责“再来一组练到对”，提供针对性重复训练（偏“行为矫正”）
- 两者组合：LLM让你**知道**错在哪，错题回顾让你**练到**不再错

### 6. 抖音直播模型

**直播核心结构**：
1. **开场钩子**：“你英语慢，不是不会，是你不会删信息”
2. **现场实验**：展示同一长句的两种读法
3. **核弹级效果演示（双屏对比）**：
   - 左屏：完整长难句，按正常语速通读，计时
   - 右屏：同一句子，干扰词灰字+删除线，只读黑字主干，计时
   - 定格：**“4.2秒 vs 1.1秒”**
4. **核心结论**：“英语不是翻译问题，是信息筛选问题”
5. **转化**：9.9元体验 / 29元系统训练

#### 11.1 支付接入方案
| 场景 | 支付方式 | 备注 |
|------|----------|------|
| 公众号H5 | 微信JSAPI支付 | 需认证服务号，openid换支付参数 |

#### 11.2 支付流程
```
用户点击“29元解锁”
    ↓
前端调后端 `/api/create-order`
    ↓
后端生成订单（状态：待支付），调用微信支付统一下单
    ↓
返回 prepay_id → 前端调起微信支付 SDK
    ↓
用户输入密码 → 支付成功 / 失败
    ↓
微信异步通知后端 `/api/pay-callback` → 订单状态改为“已支付”
    ↓
用户权益生效（paid=1, paid_expire_at=当前+7天）
    ↓
前端轮询或等待回调 → 跳转到训练首页
```

#### 11.3 掉单处理
- 前端轮询：支付页面保留，每3秒调用 `/api/check-order` 查询状态，最多轮询60秒
- 超过60秒仍未确认 → 提示“支付已受理，请稍后刷新查看”（微信回调可能延迟）
- 用户可点击“已完成支付，刷新状态”手动触发查询

#### 11.4 退款策略
- 购买后24小时内未使用 → 支持自动退款
- 已开始训练 → 原则上不退，但保留客服手动退款入口
- 退款调用微信支付退款API，同步关闭用户权益

### 12. 句子管理系统

#### 12.1 MVP方案：JSON文件管理（最简单，零后台开发）

在代码仓库中维护 `sentences.json`：
```json
[
  {
    "id": 1,
    "content": "The boy who was running in the park yesterday suddenly stopped because he saw something strange.",
    "difficulty": "中等",
    "core_indices": [0, 1, 5, 6, 8, 9],
    "tags": ["定语从句", "状语从句", "介词短语"],
    "analysis": {
      "subject": "The boy",
      "predicate": "stopped",
      "modifiers": ["who was running in the park", "yesterday", "because he saw something strange"]
    }
  }
]
```
- `core_indices` 是按空格分词后的索引数组，标注哪些词**保留**（当前仅英文，按空格分词；后续扩展中文时改用字符级索引或引入分词库）
- 运营人员手工标注，遵循第4节的标注规则
- 前端加载时解析 JSON，与用户删词结果对比

#### 12.2 长期方案（二期）
- 开发运营后台（可复用H5相同技术栈）
- 句子CRUD + 标准压缩版本标注工具
- 标注工具：选中单词后标记为“核心/干扰”，实时预览压缩效果
- 批量导入功能（支持从Excel/TXT导入）

#### 12.3 句子库维护机制
| 频率 | 动作 | 负责人 |
|------|------|--------|
| 上线前 | 录入100-300句，完成标注 | 运营 |
| 每周 | 补充10-20句新句子 | 运营 |
| 每月 | 分析训练数据，淘汰低效句子 | 运营+产品 |

难度分级标准：
- **简单**：10-15词，1个从句以内，无高级词汇
- **中等**：15-25词，1-2个从句，含常见从句类型
- **高考真题**：20-35词，2-3个从句，选自历年高考阅读理解

### 13. 异常处理与边缘情况

| 场景 | 处理方案 |
|------|----------|
| **页面刷新** | 训练中的进度自动保存到 localStorage，刷新后恢复上一次的训练状态；提交后的数据已实时保存到后端，刷新后展示“已完成”状态 |
| **网络断开** | 训练交互（点击删词）纯前端，不影响；提交时检测网络，无网络时缓存到 localStorage，恢复网络后自动提交 |
| **句子超长** | 超过30个词的句子分行展示，按意群（从句边界）换行，确保每行展示完整意群 |
| **操作失误** | 点击已灰字的词可恢复（两态切换天然支持撤销）；重置按钮一键重置当前句子 |
| **极速操作** | 有人可能1秒内完成——计时器从句子展示开始计时，到点击“提交”结束，不限制最小用时 |
| **空提交** | 用户把所有词都标记为干扰（全灰）→ 提醒“请至少保留一个核心词” |
| **空保留** | 用户什么都没标记（全黑提交）→ 正常计算，删除正确率=0%，评分低但不会报错 |
| **LLM API故障** | 回退到规则模板生成反馈，前端展示不区分来源，用户无感知 |
| **iOS/Android差异** | 点击交互使用 `click` 事件而非 `touch` 事件（兼容性更好）；iOS Safari安全区域（safe area）适配 |
| **微信内置浏览器** | 页面宽度适配 375px-414px；iOS微信页面下拉有弹簧效果，需防止误触；使用 WeixinJSBridge 唤起支付 |
| **低端手机** | 纯前端交互，无动画库，CSS动画使用 transform/opacity 避免重绘 |
